# User Stories

This document contains user stories for the **Smart Clinic** application, structured for **Admins**, **Patients**, and **Doctors**.  
Each story follows the standardized template for clarity and consistency.

---

## Admin User Stories

### User Story 1
**Title:** Admin Login  
**As an** admin, **I want** to log into the portal with my username and password, **so that** I can manage the platform securely.  
**Acceptance Criteria:**
1. Admin can log in using valid credentials.  
2. Authentication errors are displayed for invalid credentials.  
3. Successful login redirects the admin to the dashboard.  
**Priority:** High  
**Story Points:** 3  
**Notes:** Use password encryption for secure authentication.

---

### User Story 2
**Title:** Admin Logout  
**As an** admin, **I want** to log out of the portal, **so that** I can protect system access.  
**Acceptance Criteria:**
1. Admin can log out from any page.  
2. Session is terminated securely upon logout.  
3. Redirects to login page after logout.  
**Priority:** Medium  
**Story Points:** 2  
**Notes:** Ensure proper session invalidation.

---

### User Story 3
**Title:** Add Doctor to Portal  
**As an** admin, **I want** to add new doctors to the system, **so that** they can manage appointments and patient consultations.  
**Acceptance Criteria:**
1. Admin can enter details such as name, specialization, and contact info.  
2. System validates all mandatory fields.  
3. A success message appears when the doctor is added.  
**Priority:** High  
**Story Points:** 5  
**Notes:** Doctor credentials should be autogenerated.

---

### User Story 4
**Title:** Delete Doctor Profile

**Acceptance Criteria:**
1. Admin can select and delete a doctor profile.  
2. Confirmation prompt appears before deletion.  
3. Doctor record is removed from the system.  
**Priority:** Medium

---

### User Story 5
**Title:** Run Stored Procedure for Monthly Appointments  
**As an** admin, **I want** to run a stored procedure in MySQL CLI, **so that** I can get the number of appointments per month and track usage statistics.  
**Acceptance Criteria:**
1. Admin can execute a stored procedure from the system or CLI.  
2. Procedure returns accurate monthly appointment counts.  
3. Data is displayed in tabular format or exported as a report.  
**Priority:** Medium  
**Story Points:** 4  
**Notes:** Stored procedure should use optimized SQL queries.

## Patient User Stories

### User Story 1
**Title:** View Doctor List Without Login  
**As a** patient, **I want** to view a list of doctors without logging in, **so that** I can explore options before registering.  
**Acceptance Criteria:**
1. The doctor list is accessible publicly.  
2. Each doctor entry shows name, specialization, and location.  
3. Restricted actions (e.g., booking) require login.  
**Priority:** Medium  
**Story Points:** 2  
**Notes:** Cache public data to reduce load.

---

### User Story 2
**Title:** Patient Signup  
**As a** patient, **I want** to sign up using my email and password, **so that** I can book appointments.  
**Acceptance Criteria:**
1. Signup requires unique email and strong password.  
2. Successful registration stores user info in the database.  
3. Email verification is sent post-registration.  
**Priority:** High  
**Story Points:** 3  
**Notes:** Add password strength validation.

---

### User Story 3
**Title:** Patient Login  
**As a** patient, **I want** to log into the portal, **so that** I can manage my bookings.  
**Acceptance Criteria:**
1. User credentials are validated.  
2. Login redirects to the patient dashboard.  
3. Failed login shows an appropriate message.  
**Priority:** High  
**Story Points:** 2  
**Notes:** Use JWT or session-based authentication.

---

### User Story 4
**Title:** Book an Appointment  
**As a** patient, **I want** to book an hour-long appointment with a doctor, **so that** I can consult with them.  
**Acceptance Criteria:**
1. Patient can select doctor, date, and time slot.  
2. Booking fails if slot is unavailable.  
3. Confirmation message and email are sent on success.  
**Priority:** High  
**Story Points:** 5  
**Notes:** Integrate with calendar API for scheduling.

---

### User Story 5
**Title:** View Upcoming Appointments  
**As a** patient, **I want** to view my upcoming appointments, **so that** I can prepare accordingly.  
**Acceptance Criteria:**
1. Upcoming appointments are listed chronologically.  
2. Each entry shows doctor, date, and time.  
3. Past appointments are moved to history automatically.  
**Priority:** Medium  
**Story Points:** 3  
**Notes:** Display reminders 24 hours before appointment.

---

## Doctor User Stories

### User Story 1
**Title:** Doctor Login  
**As a** doctor, **I want** to log into the portal, **so that** I can securely manage my appointments.  
**Acceptance Criteria:**
1. Doctor can log in using valid credentials.  
2. System validates credentials securely.  
3. Redirects to doctor dashboard upon successful login.  
**Priority:** High  
**Story Points:** 3  
**Notes:** Implement secure authentication and session management.

---

### User Story 2
**Title:** Doctor Logout  
**As a** doctor, **I want** to log out of the portal, **so that** my data and sessions remain secure.  
**Acceptance Criteria:**
1. System terminates the session.  
2. Tokens and cookies are invalidated.  
3. Doctor is redirected to the login page.  
**Priority:** Medium  
**Story Points:** 2  
**Notes:** Ensure no data persists after logout.

---

### User Story 3
**Title:** View Appointment Calendar  
**As a** doctor, **I want** to view my appointment calendar, **so that** I can stay organized.  
**Acceptance Criteria:**
1. Calendar displays daily and weekly appointments.  
2. Each appointment shows patient name and time slot.  
3. Updates reflect dynamically when new bookings are made.  
**Priority:** High  
**Story Points:** 4  
**Notes:** Use a visual calendar component for clarity.

---

### User Story 4
**Title:** Mark Unavailability  
**As a** doctor, **I want** to mark my unavailability, **so that** patients can only book available slots.  
**Acceptance Criteria:**
1. Doctor can mark unavailable times or dates.  
2. Patients cannot book during unavailable slots.  
3. Availability updates instantly across the system.  
**Priority:** High  
**Story Points:** 5  
**Notes:** Add color-coded visual indicators.

---

### User Story 5
**Title:** Update Profile  
**As a** doctor, **I want** to update my profile with specialization and contact information, **so that** patients have up-to-date details.  
**Acceptance Criteria:**
1. Editable fields: specialization, bio, contact info.  
2. Updated data reflects immediately on patient side.  
3. Validation ensures all mandatory info is provided.  
**Priority:** Medium  
**Story Points:** 3  
**Notes:** Allow profile picture upload.

---

### User Story 6
**Title:** View Patient Details  
**As a** doctor, **I want** to view patient details for upcoming appointments, **so that** I can prepare for consultations.  
**Acceptance Criteria:**
1. Patient name, age, and history are displayed.  
2. Details are restricted to assigned doctor.  
3. Access logs are maintained for compliance.  
**Priority:** High  
**Story Points:** 4  
**Notes:** Display limited medical history only.
